## §3. Principle #3 variant-check: is Universality Test a distinct framework element?

### §3.1 Comparison against M1 (rigor protocol CORE module)

M1 tests: *"Does the framework's CORE math/logic hold up to external scrutiny?"*

Universality Test (per v2) tests: *"Does the framework produce sensible answers across extraction contexts, including boundary conditions?"*

These are related but distinct:
- M1 is meta-about-the-framework (rigor protocol testing the framework's integrity).
- Universality Test is object-level (using the framework across contexts).

**Verdict:** Universality Test is not subsumable into M1 directly. M1 of the rigor protocol is a *protocol module*, not a framework-internal test. They operate at different layers.

### §3.2 Comparison against AIT

AIT tests: *"Is this candidate cost scarcity-grounded — does it survive abundance inversion?"* — operates on individual costs.

Universality Test operates on the *framework itself* (does it apply here?), not individual costs.

**Verdict:** Distinct from AIT. The v2 "complements AIT" language positions them as different-methodology-pair.

### §3.3 The in-book-usage test (decisive)

Scan the book's case studies:
- McDowell (coal extraction) — framework applied; no explicit universality test exercised.
- Deepwater Horizon — framework applied; no explicit universality test exercised.
- Libby vermiculite — framework applied; no explicit universality test exercised.
- Social Security / pensions — framework applied; no explicit universality test exercised.
- Hypothetical Mars / asteroids — mentioned as scope illustration, but these aren't case studies; they're scope-claim examples.

**Finding:** The book doesn't RUN universality tests. It applies the framework to diverse cases and asserts in prose that this diversity demonstrates generality. No named test is exercised.

Contrast with Abundance Masking: every extraction case in the book has costs being masked by abundance — it's an active mechanism in every case, just unnamed.

**Verdict:** Universality Test names a methodology that isn't being exercised. Unlike Abundance Masking (mechanism present but unnamed in cases), Universality Test is a methodology that's neither present nor named in cases.

### §3.4 What the concept actually does in the book

The concept (generality + boundary-awareness) appears in:
1. **Scope claims in prose:** "this framework applies across extraction contexts" — handled by prose assertion.
2. **Mars/Europa illustrations:** pedagogically useful boundary-condition examples — handled by Tech Appendix passage.
3. **Framework's self-identification of limits:** "the framework reports its own limits" — this is load-bearing and should be preserved (Tech Appendix §F) but doesn't need a dedicated named test.

All three can be handled without a named "Universality Test" term.

## §4. AIT-pairing argument (detailed comparison)

### §4.1 Why Abundance Masking's pairing earned Ring 2

Abundance Masking + AIT form a **mechanism + detection-methodology pair**:
- Without the mechanism named, AIT's definition ("tests whether a candidate cost is scarcity-grounded") has no named phenomenon it's detecting.
- Adversarial reader: *"What does AIT detect?"* — must be answered with a named mechanism.

### §4.2 Why Universality Test's pairing doesn't earn Ring 2

Universality Test + AIT form a **discovery-methodology + validation-methodology pair** (per v2).

But:
- AIT's definition doesn't depend on Universality Test being named. AIT is self-contained as "inversion test for scarcity-grounding."
- The adversarial reader's question *"What does AIT detect?"* is answered by "costs masked by abundance" — doesn't require Universality Test to be named.
- Universality Test's role is about the framework's scope, not about completing AIT's definition.

**Asymmetric pairing:** AIT needs Abundance Masking to explain what it detects; AIT does NOT need Universality Test to explain what it is.

### §4.3 V2-pairing language is weaker than it appears

V2 says *"complements the Abundance Inversion Test"* — "complements" is weaker than *"AIT is the detection methodology for Abundance Masking."* The Abundance Masking definition makes the pairing explicitly load-bearing (without masking, AIT has no named object). Universality Test's pairing is categorial (both are methodologies) but not structurally required.
