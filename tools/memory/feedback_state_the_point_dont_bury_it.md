When drafting publisher-facing prose, do NOT wrap the main point in a setup
sentence before it and an explanatory tail after it. State the point and trust
the reader to feel why. Front-loading ("Imagine X... watch Y... Call it Z")
buries the beat; lead with the beat ("it has a name: Z"). After the point,
resist the explain-it-again clause ("...because the only reason it was ever
free is that something else was paying") — the reader already got it. Where a
sentence can either explain backward or push forward, push forward: a sentence
that moves the reader into the next move ("Now you start to wonder what, and
who, is carrying the cost right now") beats one that re-explains the last.

**Why:** Surfaced 2026-06-24 during the §00 book-proposal opening redline. The
author repeatedly caught me inserting a sentence or two before the main point,
which "adds length and takes away impact from the main point," and rewrote a
CIT paragraph to land the name immediately and end on a forward-driving
question. Pairs with the em-dash tic — both are LLM-prose padding habits the
author redlines out. Division of labor holds: Fable = coverage + fidelity,
author = voice ([[feedback-author-redline-final-stage]]); but apply this
proactively so the author's redline has less to strip.

**How to apply:** Draft the point first, then delete any preceding scene-set
that only announces it's coming, and any following clause that restates it.
Prefer forward-motion closers over backward-explanation closers. Related:
[[feedback-em-dash-overuse]], [[feedback-author-redline-final-stage]],
[[feedback-voice-polish-pipeline]].
