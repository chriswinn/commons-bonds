# feedback_cost_constraint_removal_default_reset

**Date:** 2026-06-02
**Status:** RATIFIED
**Origin:** Pipeline two-level reorg session 2026-06-02; codifies Amendment X.4

## Summary

When the author removes a cost constraint (tokens / time / author-attention),
**relocate the procedural baseline entirely under the new framing — do not
expand marginally from the old baseline.** The corrective is *structural*, not
*numerical*.

The failure mode this guards against: I update *intellectually* ("yes, tokens
are free now") but functionally revert to "minimum-discipline action" defaults —
the same drip-effort behavior, nudged slightly. The result is under-action
dressed as compliance: holding back from a full cascade re-fire after "ratify
all"; recommending the cautious option when zero-cost framing should pull toward
the thorough one; treating a normal action as if it needed higher authorization;
spawning fire-and-forget background work when the author wanted interactive
engagement.

## The rule

On any cost-constraint removal, ask: **"If this constraint had never existed,
what would the default behavior be?"** Adopt *that* baseline wholesale. Do not
ask "how much more can I do than before?" — that anchors on the constrained
baseline the author just lifted.

Concretely:
- Token budget lifted → the default is the most thorough correct path (full
  cascades, additional passes, hybridization, comprehensive sweeps), not "a bit
  more than the cheap version."
- The cost that *still binds* is real-world: **author attention + ratification
  cycles + clock time + real expense (e.g. fieldwork travel).** Optimize against
  *those*, not against tokens. (See the material-readiness gate's branch-(b)
  cost-justification in the pipeline doctrine: fieldwork EV is weighed in
  real-world cost, never token cost.)

## Why

Ratified 2026-06-02. Empirical anchor: the Atlantic Ideas package session
(2026-05-28 → 2026-06-01) hit this repeatedly after cost constraints were
removed (handoff §1 Lesson 1). The originating session updated intellectually
but reverted to drip-effort defaults; author feedback had to redirect it
multiple times, and the work was eventually re-scoped into a proper handoff.
The 2026-06-02 reorg session was explicitly instructed to watch for this
pattern and not replay it.

## How to apply

- At the moment a constraint is lifted, explicitly re-derive the unconstrained
  default and state it, rather than silently continuing at the old altitude.
- Treat "marginal expansion from the prior baseline" as a red flag that you've
  anchored on the removed constraint.
- Do not decline marginal-improvement work on the removed-constraint's grounds
  (per `feedback_token_cost_not_a_constraint.md`).
- Keep optimizing the constraints that *do* bind — author attention, clock,
  real-world expense — which often means concentrating author-facing decisions
  and offloading mechanical work to (foreground, per session posture) chips.
