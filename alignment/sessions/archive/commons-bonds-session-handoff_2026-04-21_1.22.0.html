Commons Bonds operating core (canonical — update only on protocol revision)

AIT positioning

The Abundance Interference Taxonomy is the framework's epistemic core. Layers are organizational scaffolding, not load-bearing structure. Layer additions, mergers, or renames are organizational revisions, not framework-integrity events. Canonical statement: C9 patch §3.

Two-path rigor test — apply before ratifying any architectural claim

  • Path 1 — Allocation symmetry: Is the structural value-capture / cost-bearing asymmetry present?
  • Path 2 — Shield absence: Is the distance condition (the shielding mechanism) absent?

Both paths must confirm. Structural allocation is the mechanism of cost severance. Distance is the shielding condition.

Merger gate — apply before ratifying any new or renamed layer

Demonstrate bilateral independence with cases in both directions: a case where Layer A costs occur without Layer B costs, and vice versa. If independence fails in either direction, merger is indicated.

Primitiveness gate — apply before ratifying any new layer

The layer's revealed costs must not decompose cleanly into compositions of existing layers. If they do, the candidate is a cross-layer concept, not a new layer.

Operating discipline

"Push harder so we only rewrite once." Applies to layer set, protocol, AIT positioning, and chapter framing — not just wording.

Methodology notes (this session)

  • Preferences architecture decision: CB operating core lives in both preferences (always-on convenience) and handoff files (self-contained, portable). For non-CB projects: handoff only, no preferences bloat. Handoff-as-self-contained-document is the more robust general design — preferences is a convenience layer on top.
  • Two-template conclusion: One CB-specific template (operating core pre-populated), one generic template (operating core as fillable block). A third "public" variant is redundant — the generic template already serves that function.
  • Boat evaluation rule (new): Ruling out a specific boat at a specific condition/price is not ruling out a model class. Evaluation criterion: if (purchase price + estimated repair cost) represents compelling total value, the boat is a candidate regardless of whether repairs exceed purchase price. A $500 boat needing $2,000 of work = $2,500 all-in; evaluate on all-in value, not the ratio.
  • Memory editing sequencing: Do memory editing at the very start of a session, before any project work, so corrected context is active throughout.
  • Session handoff versioning across project types: CB handoffs continue the v1.##.# sequence. Non-CB handoffs use their own independent sequence (e.g., sailboat-handoff-YYYY-MM-DD_v1_0_0.html).
