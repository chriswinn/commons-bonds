Commons Bonds — Workstream Alignment Companion

Portable reference for any workstream session · Last revised April 19, 2026 · Upload alongside the active workstream file to preserve project-wide alignment

Purpose of this document. The Commons Bonds project has accumulated architectural decisions, scope boundaries, retired vocabulary, cross-reference conventions, and known drift patterns across many sessions. Individual workstream files don't always carry this context. This companion travels with whichever workstream is active and gives a new session everything it needs to stay aligned without re-litigating settled decisions. Read it first. Then read the workstream.

1. Project architecture (Option C, canonical)

Two pillars
  1. Cost Severance — naming the mechanism by which extraction separates value capture from obligation.
  2. Residual Commons Value (RCV) — pricing what was previously unpriceable, via the eight-tier decomposition.
Two methodologies
  1. Abundance Inversion Test — discovery. Strip an abundance layer; what becomes visible is a tier.
  2. Universality Test — validation. If the framework applies on Mars and on an asteroid, the mechanism is structural, not ideological.
Book spine: Mechanism → Math → Universal Validation → Actionable Policy.

2. Two-book split (Decision D, April 19, 2026)

Book One = Commons Bonds. Framework, non-partisan, minimal legal exposure, near-term (Noema → book proposal). Vocabulary establishment is the deliverable. Legislative passage is explicitly not the goal.

Book Two = The Subsidy Economy. Applied / policy / advocacy. Notes-only for now. Pending 3–5 years after CRNA / PhD / institutional affiliation or coauthor attachment.

Success metric (Book One): "If in ten years a labor lawyer uses 'severed cost' in a brief and the judge doesn't need it explained — and the author never knows it happened — the book succeeded."

Scope discipline test (apply to any content being added to Book One)

  1. Does this sharpen the framework, or expand scope? (Sharpen = add. Expand = probably skip.)
  2. Does it help a reader recognize the mechanism in their own situation?
  3. Does it add vocabulary the discourse needs, or duplicate vocabulary already established?
  4. Does it require Book One to make claims that would be stronger if held for Book Two?
  5. Does it increase legal or political exposure without proportionate utility increase?

When in doubt, trim. Book One's job is to make the invisible visible and hand the vocabulary to the discourse. Anything beyond that job risks weakening the book's ability to do its actual job.

Book Two tells — content that should move to /_One Day Maybe/Book-Two-Subsidy-Economy/

  • Named legislation, named agencies, named coalition partners, named executives
  • Legislative pathway tables (local/state/federal/international)
  • "Entering wedges" tactical language
  • Phase 1 / Phase 2 / Phase 3 implementation timelines
  • Political risk mitigation matrices
  • Specific institutional adoption targets (e.g., "OECD adopts X by year Y")
  • Global subsidy aggregation beyond the single "~$10–15T annually" signal sentence
  • Industry-specific exposés or named-beneficiary analysis

8. How to use this document in a session

  1. Open whichever workstream file is active for this session.
  2. Open this alignment companion alongside it.
  3. Read Section 1 (Option C architecture) and Section 2 (two-book split) first — these are the non-negotiables.
  4. Read the relevant row in Section 5 (per-workstream flags) for the active workstream.
  5. Check Section 4 (known alignment issues) to see whether the current session intersects any drift pattern.
  6. Proceed with the workstream.
  7. Before closing the session: run the workstream's own "Check before ending the session" list. If the workstream lacks one (WS11–17), use Section 6's template to add one.
  8. Add an entry to Section 10 (revision log) capturing what was done, which alignment issues closed or newly surfaced, and which per-workstream flags changed. Update the scorecard at the top of Section 10 to match.

9. What this document is not

  • Not a replacement for the workstream files. Each workstream contains its own detailed scope, file references, and deliverables. This document is meta-level alignment only.
  • Not a decision record. WS01 (Master Decisions) is canonical for decisions. If this document ever conflicts with WS01, WS01 wins.
  • Not permanent. As WS11–17 are retrofitted to WS02–10 format and alignment issues resolve, this document's contents will shrink. Revise when issues close.
  • Not exhaustive. New drift patterns will emerge. Add them to Section 4 when they do.

When to log

  • Any session that completes or advances a workstream.
  • Any session that modifies this companion document (e.g., closing an issue in Section 4, adding a new drift pattern, retrofitting a WS11–17 file).
  • Any session that discovers a new cross-reference bug or scope issue.
  • Any session where a decision is made that changes the Book One / Book Two scope line. (Also mirror this in WS01 Master Decisions.)

When not to log

  • Reading sessions where no file is modified.
  • Purely editorial passes on chapter prose that don't touch workstreams or alignment state.
  • Satellite projects in /_One Day Maybe/ (those have their own notes).

How the log interacts with the scorecard

When a log entry closes an issue from Section 4, increment the "Alignment issues closed" count in the scorecard. When a WS11–17 retrofit completes, increment that count. When a new drift pattern surfaces, increment "New alignment issues surfaced" and also add a row to Section 4. The scorecard is a derived summary; the log entries are the ground truth. If they ever disagree, the entries win — refresh the scorecard from them.

When the log gets long

After ~30 entries, the log becomes harder to scan. At that point: (a) move older entries to a separate "Archive" section at the bottom, (b) keep the most recent ~10 entries at the top, (c) make sure the scorecard still reflects ground truth. Don't delete entries — they're the audit trail.

When this companion can retire

This document shrinks as issues resolve. When the alignment scorecard shows all 8 issues closed, all WS11–17 retrofitted, and no new issues outstanding for two consecutive months, the document's job is mostly done. At that point it can be trimmed to just Sections 1 (architecture), 2 (two-book split), and 10 (log). Sections 3–7 can be archived. Sections 4 (known issues) and 5 (per-workstream flags) collapse naturally as entries close.
