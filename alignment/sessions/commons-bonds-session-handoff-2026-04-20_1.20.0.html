Notes on methodology / process (established this session)

  • Routing before testing. Classify the claim (tier-level, architectural, stakeholder-facing, novelty, applied-policy) before running tests. Union when multiple categories apply. Codified in protocol §3.
  • Failing test is a finding, not a veto. A failed test surfaces a gap; response is refine, accept with documentation, or reject. Silence on a failed test is the illegitimate outcome.
  • Two-path first. For architectural claims, run two-path first (Paths 1 and 2 at all relevant scales and environment types). AIT and ten-critic are cross-checks, not primary. This is what the C6 v1.0.0 → v2.0.0 rerun taught.
  • Off-Earth exemplar is not speculation. The framework's universality claim covers these cases. Exercising the tests against engineered-habitat readers now catches failure modes that would otherwise only appear in application.
  • "And other" pattern for class-defining lists. Prevents name lock-in. Apply to revealed-costs lists, manifestation paragraphs where a class is being named, glossary entries with multi-instance definitions.
  • Naming convention (unchanged): Chapter_#_Title_Draft.md = prose draft. Chapter_#_Title_GuidanceDoc.md = guidance doc. Decision memos: commons_bonds_{topic}_{v#_#_#}.html or .md. Patches: c{N}_{topic}_patch.md. Methodology docs: commons_bonds_{topic}_{v#_#_#}.md. Handoffs: commons-bonds-session-handoff-YYYY-MM-DD_{v#_#_#}.html.
