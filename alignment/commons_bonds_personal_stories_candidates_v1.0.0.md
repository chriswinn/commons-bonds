## §0. Why this file exists

Personal stories surface organically during working sessions — often when a framework concept reminds Chris of a lived experience that demonstrates the mechanism at individual scale. Without a capture doc, these stories risk being lost between sessions. This file is the index; full content lives in source files.

**Source-preservation discipline (per WP#10 + 2026-05-04 articulation):**

| Layer | Contents | Discipline |
|---|---|---|
| **Source files** (`research/story-drafts/<file>`) | Full and complete story content. Personal data intact. Grows over time as additional quotes / email chains / Zoom transcripts accumulate. | Never shortened. Touch only to polish prose or add new material. |
| **Index** (this file) | Status, source-file path, one-line summary, gate flags, key decisions. ~5-10 lines per entry. | Cross-reference at full project-pattern brevity. |
| **Publisher-facing chapters** (`manuscript/chapters/*`) | Mines source files for usable content. Personal-data scrubbing + consent / NDA / anonymization happen at this layer, not in source. | Per WP#8 + Pattern 2 register. |

Framework-mapping analysis lives in chapter GuidanceDocs at integration time, not here.

## §2. Operating discipline

### §2.1 When to capture a candidate

When a personal story surfaces during working sessions that (a) demonstrates a framework mechanism at individual scale, (b) has cross-chapter placement potential, or (c) the author wants to remember for later drafting. Capture immediately — don't wait for the next session.

Capture pattern:
1. Drop full content into `research/story-drafts/<descriptive-name>_<date>.md` (source layer; never shortened thereafter).
2. Add a thin index entry to §1 above (status + path + one-line summary + gate flags + placement decision if known).
3. Framework-mapping analysis happens at chapter integration time in the relevant chapter's GuidanceDoc, not here.

### §2.2 Per-project-memory integration

Per `project_personal_stories_drafting.md`: when 3–5 personal-story drafts exist, the testing suite evaluates per-story (M3 + M4 + M5 + M9 + M11). Path K integration depth gets revisited at that point. This file's index supports that workflow.

### §2.3 Source-preservation + scrub-on-promotion discipline (2026-05-04)

Per author articulation 2026-05-04 + WP#10:

- **Source files are canonical and never shortened.** Personal data intact. Touch only to polish prose or add new material (additional quotes, follow-up interviews, email chains).
- **Scrubbing happens at promotion, not at source.** When chapter prose absorbs source content, anonymization / NDA / consent / privacy decisions are applied to the chapter prose; source file stays full.
- **Author override allowed.** Specific content the author explicitly does not want preserved at any layer can be deleted entirely (Candidate #4 is the precedent).

### §2.4 Privacy + consent gates

Per Ch 1 GuidanceDoc §Pre-drafting gates: ex-wife medical privacy, parents' privacy, Hampton community observations, and CEO-era NDA restrictions apply to any personal story. Each candidate's gate flags field captures relevant issues.

**2026-05-01 ex-wife consent decision:** assume complete rejection of consent from the ex-wife. All details of her and her medical emergency are dropped from publisher-facing scope. Healthcare-end-of-life bridge re-anchors on parents + nursing-rotation observation. Affects Candidates #4 (DELETED) and #5 (PARTIAL REVISION).
