# Named-subject consent tracker

**Updated:** 2026-05-28
**Status:** PROPOSED (initial creation; verification gaps marked `(verify)`; author ratification pending)
**Canonical policy reference:** [`tools/memory/feedback_named_subject_consent.md`](../../../tools/memory/feedback_named_subject_consent.md) (ratified 2026-05-09; public-record exception added 2026-05-12; Phat worked example 2026-05-20)
**Purpose:** Single source of truth for "where are we with subject X today?" queries across the full publisher-facing portfolio (10 chapters + 10 essays + 2 op-eds + book proposal).

This file is a **READ AGGREGATOR** — it reflects state captured from policy memory + per-essay README named-subject inventories + per-subject outreach folders. It does NOT set policy or change consent state for any subject; updates flow FROM those sources TO this tracker, not the other way around. Contradictions surface to the author for resolution rather than being silently reconciled here.

## How to use this tracker

- **"Where are we with subject X today?"** — look up the subject in §A–§F below. Each row is one subject; the columns capture working-draft state, outreach status, restoration condition, and which artifacts reference the subject.
- **Pre-submission named-subject check** for any essay/op-ed/chapter heading to publication: see §H. The check gate is "for every named subject in the artifact's named-subject inventory, the tracker row shows OK-to-publish (consent landed, courtesy-notify executed-or-best-effort-complete, or anonymized-safely)."
- **Cross-essay consolidation** — subjects appearing in multiple artifacts (Colden, Darity, Mazzucato, Phat, Biggie, the author's grandfather) carry one row that lists every artifact citing them. The row captures the SUBJECT'S state; per-essay reference lists tell you where the citation lands.
- **Verification gaps** — rows marked `(verify)` need follow-up; a future session or the author should fill those in from the canonical sources listed in §G.

## How to update this tracker

1. **Trigger events that require an update**: new subject named in any publisher-facing artifact; consent landed; courtesy-notify executed; interview conducted/declined; subject anonymized or restored; subject's living/deceased status changes; outreach folder created/updated.
2. **Update path**: change the subject's row in the relevant table below. If the change crosses policy classes (e.g., consent lands → row moves from §A to §C), move the row to its new section. Note the change in the "Last activity" column.
3. **Do NOT change consent state here.** This tracker reflects state captured elsewhere. If you find a contradiction between the tracker and a subject folder / essay README, surface it to the author rather than overwriting one side.
4. **Index sync** — when a new subject lands a folder under `research/outreach/subjects/`, add the row to §A/§B/§C/§D as appropriate. When a folder is archived, leave the row but mark "archived" in last activity.

## §H. Pre-submission named-subject check

Before submitting any publisher-facing artifact (essay, op-ed, chapter for proposal sample, etc.), execute this gate:

1. **Identify named-subject inventory** for the artifact. Source: the artifact's own README §"Named-subject inventory" (essays + op-eds) OR the chapter's section frontmatter / inline HTML comments (chapters).
2. **For each subject in the inventory, look up their row in this tracker** (§A–§F above).
3. **Verify the subject's row shows OK-to-publish** per its class:
   - §A (living-pending-consent) → row shows EITHER signed consent landed (restore name) OR anonymization holds (submit anonymized; consent pursuit continues at relationship-pace).
   - §B (deceased-courtesy-notify) → row shows EITHER courtesy-notify executed (subject's family notified) OR 30-day good-faith effort completed without success (proceed with naming under "unreachable after reasonable effort" branch).
   - §C (public-record on-record speech) → row shows citation-accuracy notify packet either fired pre-submission OR scheduled to fire pre-submission as part of the submission window.
   - §D.1 (interviewed subjects) → row shows post-interview attribution-review materials sent to subject AND written approval received for direct quotes + substantive paraphrases.
   - §D.2 (published-work citations) → no per-submission check beyond standard citation-verification (Pass 3.1 fact-check pass covers this).
   - §E (deceased / historical) → no consent check; verify citation accuracy via Pass 3.1.
   - §F (author family) → author has confirmed naming for the artifact.
4. **If any row shows a gate not satisfied**, the artifact does NOT proceed to submission. Surface the gap to the author for resolution: either complete the consent action OR anonymize the subject in the artifact.
5. **Update the artifact's submission checklist** in its per-essay/op-ed README with one line per named subject showing tracker-verified-OK-to-publish status, dated.

This check is a **gate**, not a guideline. The cost of a missed consent / un-executed courtesy-notify shipping in a publisher-facing artifact propagates from essay → book → reputation (per the established no-invented-factual-claims discipline + the asymmetric reputational cost of correcting a published miss vs. catching it pre-submission).
