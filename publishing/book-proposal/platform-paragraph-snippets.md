# Platform-paragraph snippets (per-venue ready-to-paste lines)

**Date drafted:** 2026-06-09 · **Date modified:** 2026-06-09

**Purpose.** Pre-drafted lines per Wave 1 venue, in three status states each, so the author can swap them into [03_author-platform.md](03_author-platform.md) (and the platform paragraph of any active query letter) without restructuring the surrounding prose. Anchored from `03_author-platform.md` via the `PLATFORM-PARAGRAPH-UPDATE-ANCHOR` comment.

**Discipline.** Per `tools/memory/feedback_named_subject_consent.md` + standard cite-fluency. Soft "under consideration" claims may be cited only within the venue's published response window — never speculative, never beyond the window. Bylines + URLs are added only after the piece publishes.

---

## How to use

1. Locate the current state for each venue in the live tracker (per cascade-plan v2 §2 + `publishing/essays/_pipeline/decisions-log.md`).
2. Pick the matching status-state snippet below — (a) under consideration, (b) forthcoming, (c) published.
3. Substitute `[DATE]`, `[MONTH YYYY]`, `[BYLINE-URL]`, `[TITLE]` per current state.
4. Paste into the long bio's insertion site per the anchor comment in `03_author-platform.md` (typically as the second sentence, after "former healthcare-IT executive...firm.").
5. If multiple venues are live simultaneously, prefer the **two strongest current claims** in the bio paragraph — agents reading the bio scan for two anchors, not five. Use the strongest active claim from each pipeline (one essay-pipeline, one op-ed if active) and leave the rest for the Placements / Soft-clips tables.
6. Update the Placements table below the bio in lockstep.

**Render rule.** Status-state (a) is the only state that should appear in a bio line as a phrase ("under consideration at *Aeon*"). State (b) and (c) take a noun-phrase or sentence form ("Recent essays in *Boston Review* and *Phenomenal World*"; "His essay 'The Mask of Abundance' is forthcoming in *Aeon* in [MONTH YYYY]").

## Combination-line patterns (for when two-three placements are live simultaneously)

When two or three venues are active at once, the bio line gets crowded if every venue is named individually. Use combination patterns:

**Two essays, mixed states:**
> ...His essay "[TITLE]" appeared in *Boston Review* in [MONTH YYYY], and related work is currently under consideration at *Aeon* and *Phenomenal World*.

**Three essays, all published:**
> ...Recent essays have appeared in *Boston Review*, *Phenomenal World*, and *Aeon* ([YEAR]).

**Three essays, mixed states:**
> ...Recent essays have appeared in *Boston Review* and *Phenomenal World*, with additional work forthcoming in *Aeon*.

**One published + one op-ed published:**
> ...His essay "[TITLE]" appeared in *Boston Review* in [MONTH YYYY]; an op-ed on Norway's sovereign-wealth architecture appeared in [VENUE] in [MONTH YYYY].

## Sentinel queries to run before each Pipeline A wave fires

Before each Pipeline A Query Wave goes out, run these checks to ensure the platform paragraph is current:

1. **Cascade-plan `§2 Wave 1` table** — read the current decision-state for every Wave 1 venue. Match the snippet state to the table.
2. **`publishing/essays/_pipeline/decisions-log.md`** — confirm any acceptance / decline letters since the last update.
3. **`publishing/essays/_pipeline/june-2026-submission-schedule.md`** (and successor docs) — confirm submission-window currency for venues not yet submitted.
4. **`publishing/op-eds/<slug>/`** directories — check whether any news-peg op-eds have fired since the last bio refresh.
5. **`publishing/blurbs/blurb-network-register.md`** — check whether any pre-publication blurb commitments have landed (the bio long version may eventually add a "with pre-publication endorsements from..." line if multiple blurbs commit).

If a Wave fires while a soft-clip claim sits beyond the venue's published response window, downgrade or drop the claim before sending. Stale soft-clips are worse than no soft-clip.
