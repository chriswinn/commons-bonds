# Workstream 09 — Old-Material Rigor Check

Commons Bonds project · Apply current rigor to working documents that predate Option C and the villain-analysis landing

**Prerequisites:** Workstream 01 (Master Decisions), Workstream 02 (Glossary Reconciliation). Can parallelize with Workstream 08 and 10.

**Option C quick reference:** Two pillars (Cost Severance, RCV) + two methodologies (Abundance Inversion Test, Universality Test). Retired: V\_civ, Solidarity Architecture, Departure Liability, ESG.

## What this workstream does

Audits three working documents that were created before Option C was finalized and before the April 19 villain-analysis and two-book decisions. Each document needs to be evaluated against current rigor: **update**, **retire**, or **replace**. None of them is the book itself; they are scaffolding, assessment, or planning documents. But if they still drive chapter revisions, they drift the project.

The three documents are:

- `Commons Bonds - Ripple Effects`
- `commons_bonds_integrated_framework.html`
- `commons_bonds_chapter_assessment.html`

## Why these specifically

All three were active reference documents in early-to-mid April, before the Option C stress-testing, before the villain analysis, and before the two-book split. Each influences how other workstreams execute. If Workstream 06 (Chapter 10) updates the Ripple Effects doc but the Integrated Framework doc still contains the old three-contributions framing, subsequent passes will inherit conflicting instructions.

## Rigor standard to apply to each

A working document survives current rigor if and only if:

1. It does not use retired vocabulary (V\_civ, Solidarity Architecture, Departure Liability, ESG) as current terms.
2. It does not frame the book's contributions as three-originals ("cost severance + RCV + solidarity architecture") — now Option C architecture + institutional legibility landing.
3. It does not treat Chapter 8 as the full-economy simulation chapter — now a single worked example + aggregation signal.
4. It reflects or is at least consistent with the eight-tier decomposition (April 19 canonical version per Decision A).
5. It reflects or is at least consistent with the two-book split.
6. It does not advise actions that would cause Book One to drift toward Book Two exposure.

Any document failing one or more of these gets a verdict: update (surgical fix), replace (full rewrite), or retire (move to `/Archives/` and remove from active reference).

## The general pattern for old material

Working documents age poorly. Each session-layer of rigor makes prior working documents less accurate. After two months of intensive development, the project has accumulated a stack of working documents that each made sense at the time and now carry inconsistent assumptions.

The book itself — chapters, glossary, Technical Appendix — stays current because each workstream updates it. The working documents do not get updated the same way, which means they drift.

**Operational rule:** Working documents older than three weeks should be evaluated before being used as reference. If the document does not survive the rigor check above, retire it. Do not patch.

## Watch items

- **Do not patch documents that should be retired.** Patching creates the illusion of currency while preserving outdated assumptions.
- **Do not update a working document without also checking whether its contents are still referenced by chapter drafts.** If a chapter draft cites the Integrated Framework doc for a derivation, update the chapter to cite the canonical source (Technical Appendix or Glossary) before retiring the Integrated Framework doc.
- **Do not delete retired documents.** Move them to `/Archives/`. They are historical record of the project's evolution and may be useful when you eventually write the Author's Note: On Wind Tunnels and AI, or when you reflect on the framework's development.
- **Do not add a fresh chapter assessment during this workstream.** The fresh assessment should happen after Workstreams 03, 05, 06, 07 are complete — otherwise it assesses a partial state.
- **Do not rely on the Session Archive's "already clean" claims without verification.** The Archive reflects a moment; sweeps can be incomplete in files that were not touched since.
